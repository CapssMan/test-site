# SkillCheck — пакет входных данных перед пилотом

Обновлено: 23 июля 2026 года.

## Назначение и граница хранения

Этот шаблон собирает только решения, которые нельзя достоверно вывести из кода: реквизиты оператора, внешнее legal/retention заключение, независимый SME verdict, точный scope очистки smoke-данных и финальное решение владельца.

Заполненную рабочую копию хранить в закрытом owner-контуре. Не коммитить сюда подписи, телефоны, домашние адреса, закрытые юридические комментарии, answer key, токены, пароли, соли, invite-коды и private diagnostics. В Git фиксируются только согласованные публичные реквизиты для `privacy.html`/`consent.html` и безопасный итог `verified`/`blocker` без секретов.

До закрытия всех обязательных секций сохранять:

- `LEGAL_PILOT_APPROVED=false`;
- `ATTEMPT_ISSUANCE_ENABLED=false`;
- `RETENTION_AUTOMATION_ENABLED=false`.

## A. Публичные реквизиты оператора

Владелец предоставляет точные значения для одновременного обновления `privacy.html` и `consent.html`:

| Поле | Требуемое решение |
|---|---|
| Полное наименование или ФИО оператора | Точное публичное написание |
| Правовой статус | ИП, юридическое лицо, физическое лицо или иной подтверждённый статус |
| Адрес оператора | Публичный адрес в объёме, утверждённом профильным специалистом |
| Email по персональным данным | Реальный контролируемый адрес |
| Канал запросов субъекта | Email или утверждённая форма без публичного раскрытия записей |
| SLA ответа | Утверждённый срок и ответственный |

Нельзя подставлять тестовый email, адрес разработчика или вымышленные реквизиты. После согласования обновляются оба HTML-документа и, если изменился смысл обработки, новая `PRIVACY_CONSENT_VERSION`.

## B. Внешнее legal/retention решение

В закрытом журнале зафиксировать специалиста, дату, применимую юрисдикцию и ответ по каждому пункту:

- требуется ли уведомление Роскомнадзора и выполнено ли оно до запуска;
- допустима ли выбранная модель хранения российских пользователей на Яндекс.Диске;
- есть ли трансграничная передача и какое основание применимо;
- цели и категории данных минимальны и совпадают с фактическим data flow;
- срок хранения утверждён отдельно для результата, отчёта, попытки/session, invitation, deletion log и backups;
- определены legal hold, ответственный за удаление и форма подтверждения уничтожения;
- утверждены канал и SLA запросов субъекта;
- передача работодателю остаётся выключенной либо оформлена отдельным основанием и отдельным согласием для точного получателя.

Итог секции: `APPROVED`, `APPROVED WITH CONDITIONS` или `BLOCKED`. Условия переносятся в owner-журнал и runbook; Codex не заменяет внешнее юридическое заключение.

## C. Независимый SME sign-off v4

Работать по `SME_REVIEW_HANDOFF.md`. Для распределённой проверки использовать отдельные role-пакеты, чтобы каждый reviewer получил только банк своей компетенции; master-книгу на 240 вопросов без необходимости не передавать. Для каждого из пяти банков записать в закрытой копии:

| Банк | Эксперт/компетенция | Проверено | Блокирующих замечаний | Итог |
|---|---|---:|---:|---|
| FA |  | 0/40 |  | `PENDING` |
| CA |  | 0/80 |  | `PENDING` |
| FPA |  | 0/40 |  | `PENDING` |
| ACC |  | 0/40 |  | `PENDING` |
| BI |  | 0/40 |  | `PENDING` |

Сохранить SHA-256 исходной и возвращённой книги, сведения о независимости/конфликте интересов и общий verdict. `READY FOR OWNER DECISION` не открывает production gates автоматически. Любое содержательное исправление выпускается новой versioned-ротацией, а не перезаписью v4.

## D. Подтверждение очистки smoke-данных

Текущее значение protected diagnostics — 9 result rows и 9 anti-retake rows. Оно не доказывает, что все строки являются тестовыми. Перед удалением владелец утверждает точный список кодов и scope каждой операции:

| Код результата | Доказательство, что это smoke | Scope `result`/`attempt` | Утверждено владельцем |
|---|---|---|---|
|  |  |  | `NO` |

Не удалять «все 9» по одному счётчику. Для каждого кода выполнить preview, проверить counts и backup scope, затем отдельное подтверждённое удаление по `DATA_DELETION.md`. После очистки проверить active stores, operational backups, reports и protected diagnostics. Ручное редактирование JSON запрещено.

## E. Финальный owner sign-off

После закрытия A–D владелец фиксирует вне Git:

- дату, ответственного и утверждённую цель пилота;
- Git commit, frontend/backend build, Apps Script deployment и consent version;
- результаты CI, Pages, health, protected diagnostics и backup readiness;
- состояние инцидентов и доступов;
- лимит первой волны: 3–5 работодателей/рекрутеров и 10–30 прохождений;
- владельцев метрик и stop conditions S1/S2;
- итог `GO`, `LIMITED GO` или `NO-GO` с условиями.

Только после `GO` сначала отдельным действием включается legal approval, затем — issuance. Оба gate не меняются одновременно с runtime rollout. Первый owner smoke после открытия удаляется тем же подтверждаемым workflow.

## Что можно передать Codex

Для продолжения достаточно безопасной структурированной сводки без подписей, закрытых заключений и answer key:

1. публичные значения секции A;
2. итог и обязательные условия секции B;
3. verdict и количество замечаний по каждому банку из секции C;
4. утверждённый точный список кодов/scope из секции D;
5. финальное решение владельца из секции E, когда остальные пункты закрыты.

Готовые тексты постановки задачи и acceptance criteria для внешних специалистов находятся в `EXTERNAL_REVIEW_BRIEF.md`.

Связанные документы: `PILOT_READINESS.md`, `PRIVACY_CHECKLIST.md`, `SME_REVIEW_HANDOFF.md`, `EXTERNAL_REVIEW_BRIEF.md`, `DATA_DELETION.md`, `OPERATIONS.md`.
