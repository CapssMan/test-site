# SkillCheck — постоянный roadmap проекта

Обновлено: 20 июля 2026 года.

Этот файл — постоянный источник приоритетов проекта. Когда пользователь пишет «давай дальше по проекту» или «давай дальше по этапу», работа продолжается с первого незавершённого этапа с наивысшим приоритетом.

## Рабочий протокол

1. Проверить актуальное состояние репозитория, ветку и рабочее дерево.
2. Сверить фактическую реализацию с этим roadmap и `PROJECT_STATUS.md`.
3. Выбрать первый незавершённый приоритетный этап.
4. Выполнить его полностью: код, локальные проверки и необходимые интеграционные тесты.
5. Не переходить к следующему крупному этапу, пока текущий не проверен.
6. Обновить `ROADMAP.md` и `PROJECT_STATUS.md`, не записывая secrets.
7. В итоговом отчёте указать этап, изменения, файлы, проверки, commit SHA, ручные шаги и следующий этап.

## Глобальная цель MVP

Работодатель выбирает профессиональный тест; кандидат вводит минимально необходимые данные и проходит тест с таймерами и античит-механиками; система рассчитывает и безопасно сохраняет результат; повторные попытки ограничиваются; успешный кандидат получает уникальный код и подробный TXT; администратор видит только обезличенную аналитику. Персональные данные не должны попадать в GitHub или публичную админку. MVP должен быть пригоден для ограниченного пилота с работодателями и вузами.

## Принципы

1. Сначала надёжность, затем новые функции.
2. Каждая функция проверяется end-to-end.
3. Персональные данные минимизируются, secrets не коммитятся.
4. Вопросы и правильные ответы не изменяются без отдельного задания.
5. Результат — предварительная оценка, а не гарантия пригодности или найма; требуется человеческая интерпретация.
6. Архитектура не усложняется без необходимости; платная инфраструктура не добавляется до подтверждения спроса.
7. Масштабирование начинается только после стабилизации MVP.

## Этап 4. Полный аудит расчёта результатов

Не менять формулы без доказанной необходимости. Проверить перемешивание ответов, количество и случайный выбор вопросов, отсутствие дубликатов, таймер, автопереход, запрет возврата, баллы, проценты, порог, badge, рекомендацию, `blockResults`, `trustScore` и соответствие TXT.

Штрафы за уходы со вкладки: 0 → 0; 1 → −3; 2–3 → −7; 4+ → −15. Итог ограничен диапазоном 0–100.

Сценарии: 100%, 0%, пограничный результат, ровно порог, штраф переводит результат в неуспешный, нет ответа, таймаут.

## Этап 5. Аудит банков вопросов

Для каждого банка программно проверить JSON, количество, уникальность ID, непустые вопросы, минимум два уникальных варианта, индекс ответа, `timeLimit`, `points`, `block`, `difficulty`, `active`, соответствие `blocks` и явные технические ошибки. Содержательные исправления сначала предложить пользователю; автоматически контент не менять. Отчёт хранить отдельно.

## Этап 6. Админ-панель MVP

Проверить пароль, отклонение неверного пароля, отсутствие пароля во frontend и mock-данных. Реальные обезличенные данные: поиск по коду, фильтры теста/статуса, сортировка, метрики, распределение, дата, badge, уходы, наличие TXT. Добавить пустое состояние, backend-ошибку, обновление, дату загрузки, frontend/backend version и мобильную адаптивность.

Админка не показывает имя, email, Telegram, fingerprint, hash, правильные ответы или полный TXT без отдельной защиты.

## Этап 7. Получение отчёта по коду

Нельзя публично отдавать персональный TXT только по угадываемому коду. До реализации согласовать минимальную безопасную модель: ручная выдача; пароль работодателя; одноразовая ссылка; employer token; доступ только через админку.

## Этап 8. UX кандидата

Проверить весь путь от главной до результата: описание, время и число вопросов, поля и согласия, загрузка, сеть, таймер, прогресс, мобильный вид, результат, код и дисклеймер. При ошибке сохранения рассчитанный результат не теряется; есть безопасный повтор отправки; двойная отправка исключена; внутренние пути и конфигурация не раскрываются.

## Этап 9. Надёжность отправки

Защита от двойного клика, `requestId`/idempotency key, повтор при временной ошибке, один код на попытку, локальный pending result, безопасная повторная отправка, таймаут, понятный статус и логи без персональных данных/secrets.

## Этап 10. Безопасность

Аудит Git history, `.gitignore`, `.clasp.json`, manifest, CORS/JSONP, XSS, escaping, JSON/formula injection, brute force, ошибки, публичность TXT, коды, OAuth и scopes, rate limiting, health abuse, мусорные загрузки, размер payload, `testId`, длины полей и серверная проверка результата.

Критический риск: frontend знает правильные ответы и считает результат. Сначала оценить риск и трудоёмкость вариантов: оставить для раннего MVP; частичная backend-проверка; backend-расчёт; выдача вопросов backend. Большой перенос — только после согласования.

## Этап 11. Юридическая и privacy-подготовка MVP

Сверить policy, согласия, фактические данные и хранение, срок, удаление, отзыв, передачу работодателю, Яндекс.Диск, Apps Script, GitHub Pages, трансграничность, контакт, 18+ и дисклеймер. Выделить заглушки оператора, email, оснований и удаления. Не давать окончательных юридических гарантий.

## Этап 12. Удаление и срок хранения

Административно найти результат по коду, удалить TXT, запись результата и при необходимости attempt без повреждения других записей. До изменения — резервная копия. Логировать только код и технический итог. Автоудаление по сроку спроектировать, но не включать без согласования.

## Этап 13. Резервные копии

Backup `results.json` и `attempts.json` перед записью, ограниченная ротация, восстановление после повреждения, JSON-валидация перед заменой, отсутствие персональных данных в публичных местах. Не создавать бесконечные копии.

## Этап 14. Наблюдаемость и диагностика

Защищённый технический статус: frontend/backend version, backend time, Яндекс.Диск, наличие properties без значений, размеры файлов, количество результатов/попыток, последняя запись и санитизированная ошибка. Административную диагностику не публиковать без защиты.

## Этап 15. Автоматические тесты и CI

JSON и структура банков, известные шаблоны secrets, ссылки на data, JS syntax, расчёт, Telegram normalization, retake date, escaping и report formatter. GitHub Actions добавлять только после проверки отсутствия secrets и платной инфраструктуры.

## Этап 16. Документация

## Этап 17. Подготовка к пилоту

Проверить пять тестов и банки, мобильный вид, сохранение, TXT, админку, retake, удаление, backup, policy, контакты, инструкции, отсутствие production-тестовых данных, скрытый smoke, monitoring и rollback. Подготовить pilot checklist.

## Этап 18. Пилот

3–5 работодателей/рекрутеров, 10–30 прохождений. Измерять завершение, время, отказы, ошибки, распределение, полезность TXT/кода и мнения сторон. Не менять порог автоматически на малой выборке и не добавлять крупные функции до обратной связи.

## Этап 19. Аналитика качества

После достаточной выборки: среднее, медиана, распределение, слишком лёгкие/сложные вопросы, частые ошибки/успехи, время, таймауты, различия по источнику/опыту, согласованность блоков и связь с решением работодателя. Не делать сильных выводов на малой выборке.

## Этап 20. Решение о следующей версии

После пилота ранжировать задачи по ценности, риску, трудоёмкости, срочности и зависимостям. Возможные направления: employer accounts, защищённые отчёты, приглашения, наборы/конструктор тестов, backend-расчёт, БД, экспорт, dashboard, уведомления, тарифы. Не начинать до пилота и анализа.

## Ограничения работы

- Не использовать `git add .`; добавлять только конкретные файлы.
- Перед коммитом показать `git status`, `git diff`, staged files и проверить отсутствие secrets.
- Не изменять банки вопросов без согласования.
- Не удалять данные Яндекс.Диска без backup или явного подтверждения.
- Не менять Web App URL и не создавать новый deployment, если можно обновить существующий.
- Не коммитить OAuth-токены, пароли, salt, Script Properties values и локальные credentials.
- Если нужен ручной шаг, дать одну точную инструкцию и ожидаемый результат.
