# SkillCheck — приватность исходников и публичная атрибуция

Зафиксировано владельцем: 23 июля 2026 года.

## Цель

После завершения MVP скрыть исходный репозиторий и уменьшить публичную связь SkillCheck с личным GitHub-профилем владельца, не останавливая работающий сайт и не создавая ложного обещания технической «защиты от копирования».

Это две независимые задачи:

1. **Приватность разработки** — закрыть Git history, backend-код, внутренние runbooks и будущие изменения.
2. **Публичная атрибуция** — использовать бренд SkillCheck, нейтральный домен и проектный контакт вместо личной страницы автора там, где закон не требует иного.

Реквизиты оператора персональных данных относятся к отдельному legal/privacy gate. Приватный GitHub не отменяет обязанностей оператора и не гарантирует, что сведения о физическом лице можно скрыть от всех, кроме государственных органов.

## Фактическая граница защиты

- Любая публичная страница передаёт браузеру HTML, CSS, JavaScript, изображения и загруженные данные. Посетитель технически может сохранить полученные файлы, даже если выключены контекстное меню и выделение текста.
- Private-репозиторий закрывает доступ к репозиторию для новых посторонних пользователей, но не отзывает ранее созданные клоны, скачивания, кэш и отделённые публичные forks.
- Текущий frontend загружает display-only JSON всех пяти банков. В них нет правильных ответов, но тексты 240 вопросов доступны браузеру и могут быть скопированы независимо от видимости GitHub.
- Закрытые ключи, production Script Properties и данные кандидатов в публичном репозитории отсутствуют. Authoritative scoring остаётся на backend.
- Адрес `capssman.github.io` раскрывает GitHub-псевдоним владельца, хотя текущие страницы не содержат ФИО, подписи автора или ссылки на репозиторий.

Следовательно, задача формулируется как **снижение доступности исходников и стоимости массового копирования**, а не как невозможность копирования сайта.

## Ограничения GitHub Pages

По официальной документации GitHub на дату фиксации:

- при переводе репозитория личного аккаунта GitHub Free из public в private опубликованный Pages-сайт автоматически снимается с публикации;
- GitHub Pages из private-репозитория доступен на GitHub Pro, Team и Enterprise;
- изменение visibility может изменить Pages URL и потребовать повторной сборки;
- публичные forks не становятся приватными: GitHub отделяет их от исходного репозитория;
- private-репозиторий не делает публичный Pages-сайт закрытым для посетителей.

Источники:

- <https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/managing-repository-settings/setting-repository-visibility>
- <https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site/about-custom-domains-and-github-pages>

## Рекомендуемая целевая схема

### Вариант A — минимальная миграция

1. Зарегистрировать нейтральный домен SkillCheck с privacy-защитой регистрационных данных, если она законна и доступна у регистратора.
2. Подключить и подтвердить домен до смены visibility.
3. Перейти на GitHub Pro.
4. Проверить сборку Pages из private-репозитория на временном контуре.
5. Переключить DNS, обновить privacy/CSP и только после успешного smoke сделать основной репозиторий private.

Плюсы: минимум архитектурных изменений. Минусы: платный тариф; frontend и вопросы всё равно доступны браузеру; инфраструктурная связь с GitHub может определяться технически.

### Вариант B — разделение source и deploy

1. Сделать основной source-репозиторий private.
2. Публиковать в отдельный public deploy-репозиторий только минимальный статический build без backend, внутренних документов, тестов и истории разработки.
3. Выдавать deploy credential только отдельному ограниченному workflow и регулярно его ротировать.

Плюсы: можно сохранить бесплатный Pages и скрыть большую часть проекта. Минусы: опубликованный build остаётся копируемым; публичный deploy-репозиторий и GitHub URL не скрываются полностью; усложняется supply-chain и rollback.

### Вариант C — усиленная post-MVP схема

1. Private source-репозиторий.
2. Нейтральный домен и выбранный после privacy/legal review hosting.
3. Public shell без полных банков вопросов.
4. Backend выдаёт только разрешённую порцию вопросов для конкретной подписанной активной попытки.
5. Rate limiting, CAPTCHA/OTP и мониторинг массовой выгрузки дополняют, но не заменяют юридическую защиту контента.

Это рекомендуемая долгосрочная схема для рейтинга и открытого кандидатского потока. Она повышает стоимость массового scraping, но не мешает человеку переписать увиденный вопрос или сделать скриншот.

## Публичное имя владельца

- На обычных продуктовых страницах допустима брендовая подпись `SkillCheck` без блока «создатель», личного профиля и личного email.
- Публичный контакт проекта следует отделить от личной почты; рабочий кандидат — `skillcheck.privacy@yandex.ru` после его создания и защиты MFA.
- Git commit author уже содержит псевдоним `CapssMan`; смена visibility не удаляет этот след из ранее полученных копий.
- Нельзя использовать вымышленные реквизиты, скрывать обязательные сведения оператора или обещать пользователям анонимного оператора.
- Точный публичный состав ФИО, статуса, адреса/места нахождения и контакта утверждается только после актуального external legal/Roskomnadzor review. До этого реальные кандидаты не допускаются.

## Безопасная последовательность миграции

1. Выбрать вариант A, B или C после MVP и определить бюджет на домен/hosting.
2. Снять проверяемый mirror/backup репозитория и зафиксировать текущий production commit/deployment.
3. Проверить public forks, releases, Actions artifacts, package/container registries и сторонние кэши; записать, что уже нельзя отозвать.
4. Проверить GitHub-профиль и commit metadata на лишние персональные сведения.
5. Подготовить новый адрес сайта до отключения старого, проверить TLS, CSP, ссылки, privacy flow и backend CORS/redirect assumptions.
6. Выполнить desktop/mobile, пять test routes, privacy/consent/admin и production negative smoke.
7. Переключить трафик; старый адрес оставить только как короткий redirect, если это поддерживается и не создаёт takeover risk.
8. Лишь после подтверждённого нового контура сделать исходный репозиторий private.
9. Повторно проверить отсутствие публичных answer keys, secrets, внутренних artifacts и персональных данных.

Нельзя сначала нажимать `Make private` на текущем GitHub Free repository: это может немедленно выключить действующий Pages-сайт.

## Критерии готовности

- основной source-репозиторий недоступен без явно выданного доступа;
- действующий публичный URL не содержит личный GitHub username;
- в опубликованном artifact нет backend, runbooks, закрытых ключей, secrets или Git history;
- frontend не выдаёт полный банк до авторизованной попытки, если выбран вариант C;
- legal/privacy документы содержат утверждённые правдивые сведения оператора и рабочий контакт;
- задокументировано, какие старые публичные копии невозможно отозвать;
- rollback проверен без раскрытия private source.

Эта миграция не входит в pre-pilot gates этапа 17 и не должна задерживать закрытый controlled pilot. Решение о варианте принимается на этапе 20 по бюджету, результатам пилота и требованиям к открытому рейтингу.
