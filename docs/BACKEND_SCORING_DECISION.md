# SkillCheck — решение по backend-scoring

Дата решения: 20 июля 2026 года.

## Контекст

Сейчас GitHub Pages публично отдаёт JSON-банки вместе с полем правильного ответа. `test.html` выбирает вопросы, перемешивает варианты и рассчитывает `rawScore`, штраф, `finalScore`, статус и детализацию ответов. Backend проверяет строгий формат и внутреннюю согласованность payload, но не имеет закрытого независимого источника правильных ответов.

Следствие: любой пользователь может прочитать ключи ответов или отправить технически согласованный сфабрикованный результат. `saveResult` не требует server-issued attempt/challenge, поэтому валидные spam writes могут расходовать Apps Script и Яндекс.Диск quota. Код результата подтверждает сохранение записи, а не независимую проверку знаний. Античит по вкладкам, таймер и клиентский 32-bit browser fingerprint не исправляют эту модель доверия.

Публичный `dev-quick` hard-disabled в production-baseline (`PUBLIC_DEV_TEST_ENABLED=false` и `assertPublicTestEnabled` в `checkAttempt`/`saveResult`). Это закрывает самый дешёвый одновопросный bypass, но не защищает обычные публичные тесты от согласованного spam.

## Решение

Текущий hardened client-scoring допустим только для разработки, демонстрации UX и внутреннего smoke. Он не должен использоваться как авторитетное основание отбора кандидатов.

До ограниченного пилота требуется как минимум авторитетный backend-scoring:

1. публичные display-банки больше не содержат `correct`/`correctAnswer`;
2. закрытый ключ ответов хранится только на backend или в приватном хранилище;
3. frontend отправляет идентификаторы показанных вопросов и выбранные ответы, а не рассчитанный итог как источник истины;
4. backend проверяет версию банка, уникальность/допустимость вопросов и самостоятельно рассчитывает все баллы, штраф, статус, блоки и TXT;
5. admin/result API показывает только серверный результат с признаком `server-verified`;
6. клиентский расчёт, если сохранён для UX, помечается как предварительный и никогда не записывается как авторитетный.
7. сервер выдаёт короткоживущий одноразовый signed attempt/invite token, без которого `saveResult` не принимает запись;
8. `questionId` становится обязательным и сверяется с закрытым versioned bank;
9. публичный поток защищается gateway rate limiting; для произвольного email lookup используются invitation/OTP/auth, при необходимости CAPTCHA.

Для открытого пилота с недоверенными участниками или заметной ценностью результата рекомендуется следующий уровень — backend question delivery: сервер создаёт попытку, выбирает набор вопросов, возвращает display-only данные без ключей и принимает ответы только для выданного attempt token.

Большой перенос не реализован на этапе 10 без явного согласования пользователя. Текущая реализация лишь честно маркирует результаты `client-reported-unverified` и блокирует грубые несогласованные payload.

## Сравнение вариантов

| Вариант | Что меняется | Защита от подделки итоговых полей | Защита от чтения ключей | Пригодность |
|---|---|---:|---:|---|
| 1. Оставить client-scoring | Только текущая валидация и маркировка | Низкая | Нет | Разработка/демо, не pilot gate |
| 2. Частичная backend-проверка | Backend сверяет арифметику/схему, но ключи остаются публичными | Средняя против простого POST-forgery | Нет | Не решает критический риск |
| 3. Авторитетный backend-scoring | Закрытый answer key, публичные банки без ключей, server-issued signed attempt и полный пересчёт на backend | Высокая | Существенная; сами вопросы остаются публичными | Минимальный baseline до ограниченного пилота |
| 4. Backend question delivery | Серверная сессия попытки, выборка и display-only выдача, attempt token, server scoring | Высокая | Высокая против массового чтения полного банка; клиент всё равно видит текущий вопрос | Рекомендуется для открытого/adversarial пилота |

### Почему частичная проверка недостаточна

Сверка `finalScore` с присланными `isCorrect` или суммой баллов предотвращает только наивную замену одного числа. Клиент контролирует весь payload, а публичный JSON раскрывает правильный ответ, поэтому злоумышленник может сформировать полностью согласованный «идеальный» результат. Усложнение подписи в статическом frontend также не помогает: любой встроенный секрет становится публичным.

### Почему серверная выдача не делает вопросы полностью секретными

Браузер обязан увидеть текст и варианты текущего вопроса, поэтому мотивированный участник может их сохранить. Цель server delivery — не абсолютная секретность, а отсутствие полного банка и ключа в одном публичном файле, привязка ответов к реально выданной попытке, ротация и возможность отзывать/версионировать наборы.

## Рекомендуемая последовательность миграции

### Фаза A — authoritative scoring baseline

- создать приватный канонический answer key по `testId + bankVersion + questionId`;
- генерировать/публиковать display-only JSON без правильных ответов;
- выдавать одноразовый короткоживущий signed attempt/invite token до начала теста и требовать его в `saveResult`;
- принимать только `questionId`, выбранный вариант/стабильный `optionId`, время и технические признаки;
- сделать `questionId` обязательным и сверять каждый ID с закрытой версией банка, количеством и уникальностью;
- пересчитывать raw score, штраф, итог, блоки, badge и status на backend;
- сравнивать клиентский preview только для диагностики, не доверять ему;
- сохранять `scoreVerification: "server-verified"` и версию алгоритма;
- добавить regression-тесты для всех банков, shuffle и старых/новых версий;
- продумать совместимый rollout, чтобы старая страница не могла писать в новый контракт.
- закрыть email-enumeration path приглашением/OTP/auth и поставить базовый gateway rate limiting; CAPTCHA использовать как anti-automation слой, а не как identity proof.

Оценка: **48–88 часов**, примерно **360–700 тыс. токенов Codex**. Диапазон включает базовый signed attempt/invite flow и защиту controlled-pilot perimeter; выбор внешнего OTP/CAPTCHA/gateway-провайдера или полноценные аккаунты могут потребовать отдельной переоценки. Режим: **ультра** для схемы доверия и миграции, затем высокий для механических тестов.

### Фаза B — backend question delivery

- endpoint создания attempt с короткоживущим непрозрачным токеном;
- серверная выборка и фиксация списка/порядка вопросов;
- display-only выдача вопроса или набора без ключей;
- проверка, что ответ относится к выданной попытке и принимается один раз;
- TTL, resume, idempotency, retake, rate limiting и безопасная очистка незавершённых attempt;
- обновление frontend, admin, TXT, диагностики и эксплуатационных инструкций;
- нагрузочные, recovery и abuse-тесты.

Оценка: **80–160 часов**, примерно **600 тыс. – 1,2 млн токенов Codex**. Это отдельный архитектурный проект; оценка не складывается механически с фазой A, потому что часть работ пересекается.

## Необходимое решение пользователя

Перед началом крупной миграции выбрать один scope:

- ограниченный пилот: согласовать фазу A как новый обязательный этап перед pilot checklist;
- открытый или высокозначимый пилот: согласовать фазу A+B с серверной выдачей;
- только внутреннее демо: временно оставить текущую модель, сохранив явную маркировку `client-reported-unverified` и запрет использовать балл как самостоятельное решение.

До такого согласования roadmap продолжается по небольшим независимым этапам, но приглашение реальных кандидатов остаётся заблокированным этим pilot gate.
