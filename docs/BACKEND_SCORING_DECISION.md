# SkillCheck — решение по backend-scoring

Дата решения: 20 июля 2026 года. Статус: архитектура 10A реализована; реальный pilot gate ожидает ротацию ранее раскрытых ключей.

## Почему перенос был обязательным

До 10A публичные JSON содержали правильные ответы, frontend рассчитывал итог, а backend лишь проверял внутреннюю согласованность присланных чисел. Пользователь мог прочитать ключи и сформировать идеальный согласованный payload. Код результата подтверждал сохранение записи, но не независимую проверку знаний.

Частичная серверная сверка арифметики не решает эту проблему: при публичном ключе клиент по-прежнему контролирует ответы и все исходные данные расчёта. Секрет в статическом JavaScript тоже не является секретом.

## Принятое решение

Для controlled pilot реализован единый scope 10A с частью ранее запланированной server-delivery архитектуры:

1. текущие публичные банки schema v2 содержат только display-поля и стабильные opaque `optionId`;
2. закрытые versioned answer keys хранятся на Яндекс Диске и читаются backend fail-closed;
3. администратор выпускает email- и test-bound одноразовое приглашение;
4. `beginAttempt` фиксирует точный набор/порядок вопросов и возвращает HMAC-signed token на 6 часов;
5. `saveResult` принимает только выданные `questionId`, `optionId|null` и справочную timing-телеметрию;
6. persistent session, а не одна подпись, обеспечивает single-use, idempotency и восстановление частичных записей;
7. backend самостоятельно рассчитывает баллы, блоки, порог, badge, статус и TXT;
8. новый результат имеет `scoreVerification: server-verified` и `scoringAlgorithmVersion: authoritative-v1`;
9. tab/timing остаются `client-reported-unverified`, не влияют на `finalScore`, а `penalty` равен нулю;
10. legacy `checkAttempt` и tokenless `saveResult` закрыты ответом `client_upgrade_required`.

Подробный контракт и storage layout: [`../ARCHITECTURE.md`](../ARCHITECTURE.md).

## Что это защищает

| Риск | После 10A |
|---|---|
| Подмена клиентских `score/finalScore/isCorrect` | Поля запрещены, итог считает backend |
| Ответ на невыданный вопрос или option | Отклоняется по server manifest/private bank |
| Повторная запись/двойной код | Persistent state + exact request/payload hash возвращают один код |
| Подмена/просрочка token | HMAC, fixed claims, constant-time compare, `exp` и session state |
| Email enumeration через `checkAttempt` | Публичный lookup удалён; ошибка invite flow нейтральна |
| Утечка текущего answer key из HEAD | Правильные ответы и комментарии удалены из display-only JSON |
| Клиентская telemetry меняет проходной балл | Не меняет; хранится и показывается только как advisory |

Это controlled-pilot perimeter, а не полноценная identity-система. Email не подтверждается OTP; внешний CAPTCHA/managed gateway и аккаунты могут понадобиться для открытого потока.

## Критическое историческое ограничение

Старые банки уже публиковались вместе с ответами в Git history. Удаление `correct` из текущего HEAD, private repository или переписывание ветки не отзывает клоны, форки и кэши. Поэтому новая серверная проверка технически авторитетна относительно private bank, но тот же старый набор нельзя считать секретным от мотивированного участника.

До приглашения реальных кандидатов требуется отдельное содержательное решение:

- рекомендуемый вариант — новая версия банка с ротированными вопросами/вариантами/ключами;
- временный внутренний smoke допустим с синтетическим email, результатом ниже порога и выключением issuance после проверки;
- просто переписать Git history недостаточно.

Ротация содержания не выполнена автоматически, потому что постоянные правила проекта запрещают менять вопросы и правильные ответы без отдельного согласования пользователя.

## Rollout и откат

- `ATTEMPT_ISSUANCE_ENABLED=false` закрывает создание публичных попыток во время bootstrap/cutover и после smoke до ротации банка.
- Private banks создаются однократно из ещё доступной legacy-версии и затем считаются immutable; существующий файл обязан exact-match исходной миграции.
- Старый Web App URL сохраняется.
- Legacy строки в `results.json` остаются видимыми, но маркируются `client-reported-unverified`; новые — `server-verified`.
- Rollback не должен снова разрешать tokenless запись или публиковать private keys.

## Следующее решение пользователя

Подтвердить отдельную ротацию банка перед реальным пилотом. Для этой задачи нужен режим **ультра**, потому что потребуется совместно менять методику, содержание, версии и закрытые ключи, не только механический код.
