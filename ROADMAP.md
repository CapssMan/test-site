# SkillCheck — постоянный roadmap проекта

Обновлено: 23 июля 2026 года.

Этот файл — постоянный источник приоритетов проекта. Когда пользователь пишет «давай дальше по проекту» или «давай дальше по этапу», работа продолжается с первого незавершённого этапа с наивысшим приоритетом.

## Рабочий протокол

1. Проверить актуальное состояние репозитория, ветку и рабочее дерево.
2. Сверить фактическую реализацию с этим roadmap и `PROJECT_STATUS.md`.
3. Выбрать первый незавершённый приоритетный этап.
4. Выполнить его полностью: код, локальные проверки и необходимые интеграционные тесты.
5. Не переходить к следующему крупному этапу, пока текущий не проверен.
6. Обновить `ROADMAP.md` и `PROJECT_STATUS.md`, не записывая secrets.
7. В итоговом отчёте указать этап, изменения, файлы, проверки, commit SHA, ручные шаги и следующий этап.

## Глобальная цель MVP

Работодатель выбирает профессиональный тест; кандидат вводит минимально необходимые данные и проходит тест с таймерами и античит-механиками; система рассчитывает и безопасно сохраняет результат; повторные попытки ограничиваются; успешный кандидат получает уникальный код и подробный TXT; администратор видит псевдонимизированную аналитику без открытых контактов. Персональные данные не должны попадать в GitHub или публичную админку. MVP должен быть пригоден для ограниченного пилота с работодателями и вузами.

Owner scope decision от 23 июля 2026 года: MVP сохраняет все пять финансовых тестов; однобанковое сокращение не выполняется. После технического завершения все 240 вопросов проходят повторный цикл качества Q1–Q4 из `docs/FIVE_BANK_QUALITY_PLAN.md`.

## Долгосрочная продуктовая цель

После подтверждения качества на пилоте SkillCheck развивается в двустороннюю платформу: специалисты самостоятельно проходят тесты, формируют управляемый профиль компетенций и по opt-in участвуют в рейтинге по профессии; работодатели ищут сильных специалистов и получают объяснимый shortlist либо проверяют собственных кандидатов.

Текущий invite-only MVP — assessment-ядро этой модели, а не конечный интерфейс продукта. Публичные аккаунты, рейтинг, employer discovery и монетизация не входят в этапы 18–20 и не должны разрабатываться до пилотных данных. Постоянные продуктовые границы и целевая последовательность зафиксированы в `docs/PRODUCT_VISION.md`.

## Принципы

1. Сначала надёжность, затем новые функции.
2. Каждая функция проверяется end-to-end.
3. Персональные данные минимизируются, secrets не коммитятся.
4. Вопросы и правильные ответы не изменяются без отдельного задания.
5. Результат — предварительная оценка, а не гарантия пригодности или найма; требуется человеческая интерпретация.
6. Архитектура не усложняется без необходимости; платная инфраструктура не добавляется до подтверждения спроса.
7. Масштабирование начинается только после стабилизации MVP.

Статус: завершён 20 июля 2026 года.

- [x] Frontend показывает точную дату следующей попытки и оставшееся число дней.

Граница модели: retake — deterrence, а не подтверждение личности. Email не верифицируется, 32-bit browser fingerprint формирует и отправляет клиент; другой email/браузер/устройство или подмена fingerprint обходят этот контроль. До пилота усиление должно опираться на server-issued invitation/attempt, а при необходимости — OTP/account.

Автотест `scripts/test-retake.js` покрывает первый запуск, совпадение по email и fingerprint, другой тест, внутренний bypass `dev-quick`, истёкшие backend- и browser-блокировки, безопасную схему `attempts.json` и точную дату. ## Этап 19. Аналитика качества

После достаточной выборки: среднее, медиана, распределение, слишком лёгкие/сложные вопросы, частые ошибки/успехи, время, таймауты, различия по источнику/опыту, согласованность блоков и связь с решением работодателя. Не делать сильных выводов на малой выборке.

## Ограничения работы

- Не использовать `git add .`; добавлять только конкретные файлы.
- Перед коммитом показать `git status`, `git diff`, staged files и проверить отсутствие secrets.
- Не изменять банки вопросов без согласования.
- Не удалять данные Яндекс.Диска без backup или явного подтверждения.
- Не менять Web App URL и не создавать новый deployment, если можно обновить существующий.
- Не коммитить OAuth-токены, пароли, salt, Script Properties values и локальные credentials.
- Если нужен ручной шаг, дать одну точную инструкцию и ожидаемый результат.
