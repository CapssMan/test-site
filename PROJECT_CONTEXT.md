# PROJECT_CONTEXT.md

## Что это за проект

SkillCheck — MVP сайта для тестирования junior-кандидатов на финансовые позиции. Проект создаётся как практическая assessment-платформа: кандидат проходит тест, система считает результат, фиксирует поведенческие риски и формирует отчёт для работодателя.

Это не учебная викторина и не академический экзамен. Основная задача — первичный screening кандидатов перед собеседованием.

## Бизнес-идея

Работодателям сложно быстро отличить сильного junior-кандидата от слабого: в резюме часто одинаковые слова про Excel, отчётность, SQL и финансовый анализ. SkillCheck должен дать работодателю быстрый фильтр:

- кого стоит звать на собеседование;
- где у кандидата сильные стороны;
- где слабые места;
- насколько результат выглядит надёжно;
- были ли уходы со вкладки во время теста.

## Целевой пользователь

### Кандидат

Junior-специалист или студент, который хочет подтвердить навыки по финансовому направлению.

### Работодатель

HR, руководитель финансового отдела, FP&A lead, риск-менеджер, кредитный департамент, small/mid business employer.

## Какие тесты планируются

Всего нужно 5 тестов:

1. Financial Analyst Junior.
2. Credit Analyst Junior.
3. FP&A / Budget Analyst Junior.
4. Accounting Junior.
5. Finance BI / Data Analyst Junior.

Сейчас реально рабочими считаются первые два:

- Financial Analyst Junior;
- Credit Analyst Junior.

Остальные три уже присутствуют в структуре, но являются черновиками и требуют полноценного банка вопросов.

## Текущая структура сайта

```text
index.html
```

Главная страница. Показывает карточки тестов и ведёт кандидата на нужный тест.

```text
test.html
```

Основной движок тестирования. Загружает JSON с вопросами, показывает вопросы по одному, считает результат и отправляет данные в Apps Script.

```text
admin.html
```

Черновая админ-панель. Может использоваться в будущем для просмотра результатов, но сейчас основная рабочая аналитика идёт через Google Sheets.

```text
data/*.json
```

Отдельные банки вопросов по каждому тесту.

```text
apps-script/Code.gs
```

Код Google Apps Script. Он принимает результаты с сайта, создаёт TXT-отчёты и пишет данные в Google Sheets.

## Текущая механика тестирования

1. Кандидат выбирает тест на главной странице.
2. Переходит на `test.html?test=<testId>`.
3. Заполняет форму:
   - имя;
   - email;
   - телефон;
   - уровень английского.
4. Сайт загружает банк вопросов из JSON.
5. Кандидат проходит вопросы по одному.
6. У каждого вопроса есть отдельный таймер.
7. Вернуться назад нельзя.
8. Варианты ответа перемешиваются.
9. Для тестов с большим банком вопросов берётся не весь банк, а часть вопросов.
10. После завершения считается результат.
11. Результат отправляется в Google Apps Script.
12. Apps Script пишет строку в Google Sheets и создаёт TXT-отчёт.

## Важные требования к вопросам

1. Вопросы должны проверять практическую логику, а не заученные определения.
2. Правильный ответ не должен быть самым длинным.
3. Все варианты ответа должны быть примерно одинаковой длины.
4. Неправильные варианты должны быть правдоподобными.
5. Нельзя использовать абсурдные варианты ответа.
6. Вопрос должен иметь только один однозначно правильный ответ.
7. Расшифровки и объяснения должны быть в `comment`, а не в тексте ответа.
8. Для junior-уровня лучше писать термины на русском, а английский или аббревиатуру давать в скобках только при необходимости.

Пример:

Плохо:

```text
DSCR ухудшился из-за падения EBITDA.
```

Лучше:

```text
Коэффициент покрытия долга денежным потоком ухудшился.
```

## Скоринг

Текущая логика:

- каждый вопрос имеет `points`;
- считается сырой результат;
- результат нормализуется в проценты;
- применяется штраф за уходы со вкладки;
- формируется итоговый балл;
- назначается плашка.

Плашки:

```text
Junior Strong
Junior Confirmed
Borderline
Not Confirmed
```

## Античит

Сейчас реализовано:

- фиксация ухода со вкладки;
- запрет копирования;
- запрет контекстного меню;
- запрет части hotkeys;
- по одному вопросу на экран;
- ограничение времени на вопрос.

Важно: полностью запретить скриншоты в браузере нельзя. Цель механики — не абсолютная защита, а повышение стоимости списывания.

## Google Sheets

Нужная структура колонок в таблице:

```text

Тест
```

Нужно убрать из таблицы:

```text
Решение
Баллы (сырые)
Всего (сырые)
```

Сырые баллы могут оставаться внутри TXT-отчёта, но не нужны в основной таблице.

## Что нельзя ломать

1. Нельзя ломать текущий дизайн сайта.
2. Нельзя удалять рабочий выбор тестов.
3. Нельзя ломать загрузку JSON-файлов.
4. Нельзя ломать отправку результатов в Google Apps Script.
5. Нельзя менять структуру вопроса без обновления `test.html`.
6. Нельзя делать правильные ответы очевидными по длине.
7. Нельзя добавлять новые тесты прямо в HTML, лучше через `data/*.json`.

## Ближайшая стратегия

Сначала довести 5 тестов до рабочего состояния. Затем начать привлекать людей, собирать первые результаты, смотреть статистику и улучшать вопросы по фактическим данным.

Приоритет:

1. Довести Credit Analyst Junior.
2. Довести FP&A.
3. Довести Accounting Junior.
4. Довести BI/Data Analyst Finance.
5. Улучшить Google Sheets и TXT-отчёты.
6. Добавить нормальную админку.
