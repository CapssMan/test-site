# SkillCheck — оценка оставшейся работы

Обновлено: 20 июля 2026 года после технического завершения этапа 10A.

Оценка относится к текущему roadmap MVP, а не к неопределённому «идеальному коммерческому продукту». Этап 10A реализовал authoritative scoring, invitations и private banks, но текущие answer keys нельзя считать снова секретными: они остаются в Git history, клонах и кэшах. Поэтому limited-pilot readiness требует этапов 11–17 **и отдельно согласованной содержательной ротации банков**. Этапы 18–20 требуют реального пилота, данных и решений, поэтому включают календарное ожидание, которое нельзя заменить разработкой.

## Сводка

- Осталось **10 плановых этапов** (`11–20`): примерно **63–114 часов**, **450–920 тыс. токенов** и **2–4 календарные недели** пилота.
- Отдельная содержательная ротация банков: **50–100 часов**, **400–800 тыс. токенов** плюс обязательная проверка профильным экспертом (SME review).
- До limited-pilot readiness (`11–17` + ротация): **93–174 часа** и **740 тыс. – 1,48 млн токенов**.
- До полного завершения roadmap вместе с ротацией: **113–214 часов**, **850 тыс. – 1,72 млн токенов** и **2–4 календарные недели** пилота.
- Production rollout/smoke этапа 10A ещё `pending`, но не считается отдельным этапом и не включён повторно в диапазоны разработки.

Это не тарифная или биллинговая гарантия. Фактический расход зависит от найденных дефектов, необходимости переделывать архитектуру, объёма live-проверок и решений пользователя.

Следующий плановый этап — **11**, рекомендуемый режим — **высокий**. Содержательная ротация является отдельным pilot blocker, требует режима **ультра** и не начинается без явного подтверждения пользователя, потому что затрагивает вопросы, варианты и правильные ответы.

Технический 10A рассчитан на controlled-pilot perimeter. Полноценные аккаунты, внешний OTP/magic-link, CAPTCHA, managed API gateway и расширенная backend question delivery остаются возможным усилением для открытого/adversarial запуска и в текущие диапазоны не включены. Ротация Яндекс OAuth-токена и проверка app-folder/least-privilege доступа входят в pilot security checklist; миграция хранилища на другой credential/API потребует отдельной оценки.

Юридическая проверка профильным специалистом, работа рекрутеров и работодателей, а также само прохождение тестов кандидатами в часы разработки не включены.

## Почему после 10A всё ещё нужна ротация

Текущая ветка больше не содержит `correct`, frontend не рассчитывает результат, а backend сверяет ответы с закрытым versioned-банком. Это устраняет доверие к клиентскому баллу, но не отзывает уже опубликованные ключи. Любой старый clone, Git history или промежуточный кэш может сохранять соответствие вопроса правильному ответу.

Поэтому реальный pilot gate требует новых либо существенно переработанных вопросов/вариантов/ключей, повторной технической проверки и SME review. Переписывание истории Git само по себе недостаточно: оно не удаляет чужие клоны и кэши. `ATTEMPT_ISSUANCE_ENABLED` должен оставаться `false` до завершения этой работы.

## Как экономить токены без потери качества

1. Использовать `ультра` для содержательной ротации/security-решений, `очень высокий` для pilot checklist и `средний`/`высокий` для механических тестов и документации.
2. Не повторять полный аудит неизменённых банков вопросов на каждом этапе; запускать готовые скрипты и показывать только итог.
3. Сохранять один этап в одном commit и не менять требования внутри этапа без необходимости.
4. Проверять production один раз после локальной матрицы, не повторяя одинаковые smoke-запуски.
5. Сохранять Apps Script + Яндекс.Диск, пока измеренные квоты позволяют; не добавлять БД без отдельного эксплуатационного обоснования.
6. Не смешивать ротацию банков с этапом 11: сначала явно утвердить содержательный scope, затем менять вопросы одним контролируемым пакетом.

Диапазоны уже предполагают такой экономный режим работы. После утверждения объёма ротации нужна короткая переоценка по числу новых/переработанных вопросов и доступности SME.
