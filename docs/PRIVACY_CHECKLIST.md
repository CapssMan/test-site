# SkillCheck — privacy checklist оператора

Обновлено: 21 июля 2026 года, этап 16.

Это инженерный контрольный список, а не юридическое заключение. Внешний специалист и оператор должны определить применимые основания, уведомления, сроки, формы подтверждения уничтожения и реквизиты до приглашения реальных кандидатов.

## Безусловные pilot blockers

До закрытия каждого пункта сохранять `LEGAL_PILOT_APPROVED=false` и `ATTEMPT_ISSUANCE_ENABLED=false`:

- [ ] в `privacy.html` и `consent.html` заполнены реальные реквизиты и контакт оператора;
- [ ] внешний legal review завершён, применимость уведомления Роскомнадзора и трансграничной передачи документирована;
- [ ] по каждой категории данных утверждены срок хранения, legal hold и ответственный за удаление;
- [ ] утверждены канал и SLA для запросов субъекта данных;
- [ ] утверждены получатели результатов и отдельное основание/согласие на передачу каждому работодателю;
- [x] банки технически содержательно ротированы выпуском v4 после исторической публикации answer keys;
- [ ] банки v4 получили независимый человеческий SME sign-off;
- [ ] Яндекс OAuth-токен ротирован и оценён на least privilege/app-folder scope;
- [x] технический этап 17 и rollback/readiness runbooks завершены с решением NO-GO;
- [ ] после закрытия остальных блокеров выполнен финальный owner sign-off с решением `go`.

Техническая готовность сайта сама по себе не закрывает эти пункты.

## Перед каждым изменением privacy flow

- [ ] описаны цель, категории данных, получатели и срок;
- [ ] минимизация подтверждена: новое поле действительно необходимо;
- [ ] privacy/consent версии обновлены согласованно с backend contract;
- [ ] старые и новые версии различимы в сохранённой записи;
- [ ] обязательное согласие не объединено с необязательной передачей работодателю;
- [ ] отказ от необязательной обработки не блокирует базовый доступ без обоснования;
- [ ] тесты legal/privacy, deletion, security и full `npm test` зелёные;
- [ ] документация удаления, backup и диагностики учитывает новую категорию.

## Перед созданием приглашения

- [ ] есть утверждённая цель и конкретный тест;
- [ ] используется минимально необходимый email; телефон не собирается;
- [ ] приглашение передаётся кандидату по закрытому каналу, bearer-код не помещается в query string;
- [ ] кандидату доступны актуальные privacy notice и отдельное versioned consent;
- [ ] подтверждение 18+ включено;
- [ ] получатель результата не включён без отдельного согласия;
- [ ] оператор понимает, что email/fingerprint hashes снижают раскрытие, но остаются данными, связанными с субъектом;
- [ ] нет дублирующего активного приглашения без документированной причины.

## Во время обработки

- [ ] доступ к Apps Script, админке и Яндекс.Диску выдан только действующим операторам;
- [ ] пароль, OAuth-токен, salts и signing secrets не передаются в чатах/issue и не хранятся в Git;
- [ ] публичный health не используется для получения storage-деталей;
- [ ] защищённая диагностика не копируется целиком во внешние системы без необходимости;
- [ ] TXT-отчёт открывается только по служебной необходимости;
- [ ] ручные выгрузки имеют владельца, срок удаления и защищённое место хранения;
- [ ] автоматические решения о найме не принимаются только по SkillCheck;
- [ ] telemetry времени/уходов со вкладки трактуется как `client-reported-unverified`, без скрытого штрафа.

## Запрос субъекта данных

1. Зарегистрировать дату, канал, тип запроса и ответственного без помещения лишних данных в публичный tracker.
2. Проверить личность пропорциональным способом; не раскрывать наличие записи по одному непроверенному email.
3. Искать запись по коду результата и допустимым внутренним идентификаторам из защищённого owner context.
4. Для удаления выполнить preview, проверить scope (`result` или `attempt`) и только затем подтверждённую операцию из админки.
5. Не редактировать JSON/TXT вручную. При частичном сбое использовать deletion recovery.
6. Проверить admin result, attempt/session/invite state, report и operational backups согласно `DATA_DELETION.md`.
7. Зафиксировать результат и способ подтверждения уничтожения в утверждённой оператором форме.

Точные юридические сроки и допустимые исключения задаёт внешний checklist, а не исходный код.

## Инцидент с данными или credential

- [ ] закрыть выдачу новых попыток;
- [ ] определить затронутые категории, период и системы;
- [ ] ротировать скомпрометированный credential; не публиковать старое или новое значение;
- [ ] проверить Apps Script executions, защищённый status, Яндекс.Диск и историю Git;
- [ ] сохранить минимальные доказательства без создания новой неконтролируемой копии ПД;
- [ ] привлечь назначенного privacy/security ответственного;
- [ ] определить применимые уведомления и сроки с внешним специалистом;
- [ ] после устранения выполнить CI, health, status и отдельное решение `go/no-go`.

Удаление потенциально затронутых данных до фиксации минимально необходимых доказательств согласуется с ответственным за инцидент и применимыми требованиями.

## Регулярная проверка оператора

Не реже утверждённого внутреннего интервала и перед каждым pilot window:

- [ ] список доступов и владельцев актуален;
- [ ] operator contacts и privacy-тексты актуальны;
- [ ] consent version совпадает с backend;
- [ ] pilot gates имеют ожидаемое значение;
- [ ] backups проверяемы, corrupt artifacts отсутствуют или разобраны;
- [ ] retention dry-run и legal hold рассмотрены, automation не включена без утверждения;
- [ ] просроченные ручные выгрузки уничтожены;
- [ ] открытые запросы субъектов и инциденты не просрочены;
- [ ] Яндекс OAuth scope и дата последней ротации задокументированы вне репозитория;
- [ ] ответственные за deployment, rollback, deletion и incident response доступны.

## Минимальный закрытый журнал решения

Для каждого pilot window оператор хранит вне публичного репозитория:

| Поле | Что фиксировать |
|---|---|
| Дата и ответственный | Кто принял решение и когда |
| Основание/цель | Утверждённая цель обработки |
| Версии | Git commit, frontend/backend build, consent version |
| Gates | Состояние legal/issuance/retention |
| Доступы | Дата последней сверки без секретных значений |
| Проверки | CI, health, protected status, backup/rollback readiness |
| Открытые риски | Legal, content, credential, incidents |
| Решение | `go`, `no-go` или ограниченный scope |

Связанные документы: `LEGAL_PRIVACY_REVIEW.md`, `DATA_DELETION.md`, `BACKUP_AND_RECOVERY.md`, `OBSERVABILITY.md`, `DEPLOYMENT.md`.
