# SkillCheck — содержательная ротация банков v4

Обновлено: 23 июля 2026 года.

## Статус

Техническая ротация пяти production-банков выполнена единым выпуском `rotation-v4-2026-07-21-r3`. Публичная часть содержит только отображаемые данные, закрытая часть с ключами хранится вне Git. Старые ключи v3 больше не соответствуют новым идентификаторам, формулировкам, вариантам и правильным ответам v4.

Это не заменяет независимую ручную проверку профильным специалистом. До внешнего SME sign-off, реквизитов оператора, legal/retention решения, очистки smoke-данных и owner sign-off решение остаётся **NO-GO для реальных кандидатов**; `LEGAL_PILOT_APPROVED=false` и `ATTEMPT_ISSUANCE_ENABLED=false` не меняются.

## Что считается полной ротацией

- у v4 нет пересечений со старыми question ID и option ID;
- формулировки вопросов, варианты и ключи созданы заново, а не переименованы;
- точные старые вопросы и варианты запрещены сборщиком;
- каждый новый вопрос сравнивается со всем закреплённым legacy baseline: максимум сходства должен быть ниже `0.78`, средний максимум банка — ниже `0.55`;
- варианты получают непрозрачные детерминированные ID и сортируются независимо от позиции правильного ответа;
- распределение фактических позиций правильных ответов ограничено: ни одна позиция не может превышать 40% банка;
- эвристики «самый длинный/самый короткий ответ» имеют отдельные fail-closed лимиты;
- публичный JSON не содержит `correctIndex`, `correctOptionId`, комментариев, объяснений или иных полей ключа;
- закрытый банк связан с публичным через `publicDigest`, а закрытые версии — с SHA-256 trust anchors в Script Properties.

## Изолированная сборка

Authoring-source и закрытые артефакты находятся за пределами Git worktree. `.gitignore` и secret scan отдельно запрещают отслеживание `*.rotation-source.json`, private-bank каталогов, rotation manifests и anchor-файлов.

Сборка выполняется только из полного набора пяти источников. Перед созданием файлов валидируются все банки; выходные каталоги должны быть новыми, внешними и не могут быть symlink/junction-подменой. Две ранние локальные ревизии были отклонены до публикации из-за побочных сигналов длины ответов и уточнений качества. Допущена только неизменяемая ревизия `r3`.

Публичное продвижение пяти JSON использует журнал и commit marker:

1. все пять файлов и manifest полностью проверяются;
2. создаются и синхронизируются временные файлы и recovery-журнал со старыми/новыми digest;
3. выполняется замена пяти targets;
4. новые digest проверяются повторно;
5. только после этого создаётся commit marker и удаляются backup/temp-файлы.

Обрыв до commit marker идемпотентно возвращает все старые банки. Обрыв после marker завершает новый набор. Повреждение оставляет журнал и блокирует дальнейшее продвижение.

## Оставшиеся ограничения

- Только Credit Analyst выбирает 40 вопросов из пула 80. Остальные четыре банка пока содержат ровно 40 вопросов, поэтому внутри одной версии нет вариативности набора.
- Порог, веса и сложность нельзя автоматически менять до пилотных данных и SME-решения.
- Исторические v3 ключи остаются фактом утечки в старой Git history/клонах/кэшах, но относятся к выведенному из эксплуатации содержанию; удаление history не требуется для безопасности v4 и не заменяет обычный secret hygiene.
- Любое следующее изменение текста, вариантов или ключа создаёт новую версию банка и повторяет private/public verification и atomic cutover.
